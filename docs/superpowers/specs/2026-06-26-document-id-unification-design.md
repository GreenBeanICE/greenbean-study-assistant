# 文档唯一标识统一设计

## 背景

当前工作区文档列表的前端 `FileItem` 同时混用了两类标识：

- 上传中的本地临时 `id`
- 后端真实文档 `documentId`

这导致同一份已持久化文档可能以两条前端记录存在：

- 上传成功后保留临时 `id`
- 页面恢复已有文档时再用后端 `doc.id` 生成一条新记录

在删除时，如果只按当前列表项的本地 `id` 过滤，就可能出现“后端已删除，但我的文档列表里仍残留一条同文档记录”的现象。

## 目标

- 已持久化文档在前端只保留一个唯一身份
- 删除文档后，列表立即移除对应文档，不依赖额外点击触发刷新
- 恢复已有文档列表时保持幂等，不因重复 effect 或重复拉取产生重复项

## 方案对比

### 方案一：仅给恢复逻辑加防重

- 在恢复列表的 effect 外层增加一次性保护
- 可以减轻 React StrictMode 下的重复追加
- 不能解决“上传后临时 id”和“恢复后真实 id”并存的问题

### 方案二：删除后强制重新拉取列表

- 删除成功后重新请求后端文档列表
- 能在多数情况下刷新掉残留项
- 只是兜底，不解决前端同一文档双身份的根因

### 方案三：统一已持久化文档使用后端真实 `documentId`

- 上传中允许临时 id 存在
- 一旦上传成功，立即把该文件项收敛到后端真实文档 id
- 恢复、选中、删除、缓存键对已持久化文档统一使用后端文档 id

推荐采用方案三，因为它直接修复标识语义不一致的根因，并且能自然约束后续缓存和列表同步逻辑。

## 设计决策

### 1. 唯一标识规则

- `FileItem.id` 继续保留为前端列表项主键
- 但对“已持久化文档”，`FileItem.id` 必须等于后端真实文档 id
- `FileItem.documentId` 继续保留，用于明确表达该文件项已绑定后端文档；其值与 `FileItem.id` 一致
- 只有“上传进行中且尚未拿到后端返回值”的文件项，允许 `FileItem.id` 为临时 id 且 `documentId` 为空

### 2. 上传流程

- 选择文件后，先创建临时文件项，供 UI 立即展示
- 上传接口返回后端文档 id 后，将该临时文件项的：
  - `id` 替换为真实文档 id
  - `documentId` 设为真实文档 id
- 以真实文档 id 作为该文档后续缓存映射的 key

### 3. 恢复列表流程

- 页面恢复已有文档时，将后端返回的每个文档映射为 `id = doc.id`、`documentId = doc.id`
- 合并到本地 `files` 时按“已持久化文档 id”去重
- 如果本地还存在同一文档的临时上传条目，应由真实文档条目覆盖，不能并存

### 4. 删除流程

- 删除时先根据当前文件项获取其后端 `documentId`
- 删除成功后，前端应移除所有与该文档对应的本地记录：
  - `id === documentId`
  - 或 `documentId === documentId`
- 同步清除该文档在 sections、blocks、units、visible units 等缓存中的对应 key

### 5. 选择与缓存行为

- `selectedFileId` 对已持久化文档统一保存真实文档 id
- `documentBlocksByFileId`、`documentSectionsByFileId`、`documentUnitsByFileId`、`visibleUnitsByFileId` 统一使用收敛后的真实文档 id 作为 key
- 上传成功后如果发生临时 id 到真实 id 的切换，需要把已生成的相关状态同步迁移到真实 id

## 影响范围

- `src/features/workspace/pages/WorkspacePage.tsx`
- `src/features/workspace/pages/WorkspacePage.test.tsx`
- 如有必要，补充与文档列表/删除相关的轻量辅助函数，但不做无关重构

## 测试设计

至少补两条前端回归测试：

1. 上传成功后，文件项 id 收敛到后端真实文档 id；随后恢复列表不会生成重复文档
2. 删除一个已持久化文档后，列表立即移除该文档，不需要再点击其他文档触发刷新

如果现有测试组织方式允许，优先在 `WorkspacePage.test.tsx` 中通过 mock `uploadDocument`、`getDocumentDetail`、`listDocuments`、`deleteDocument` 复现并验证行为。

## 非目标

- 不修改后端数据库主键或接口 schema
- 不顺手重构文件管理面板的其他交互逻辑
- 不引入全局状态库或跨页面缓存系统

## 验收标准

- 同一份已持久化文档在“我的文档”中只显示一次
- 删除后列表立即消失，无需切换其他文件触发二次刷新
- 页面恢复已有文档时不再出现重复追加
- 相关前端测试覆盖该回归场景
